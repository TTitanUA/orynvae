# План реализации MVP v2 - stage 7 Сборка черновика, разбор и прогноз

Дата: 2026-05-10
Ветка: `v2`
Область: backend draft/review/forecast services and APIs, draft version persistence, post-chapter decision workflow, frontend draft assembly/review/forecast routes.
Источник: `plans/mvp-v2-implementation-plan.md`, раздел "Этап 7. Сборка черновика, разбор и прогноз".

## Цель этапа

Превратить завершенную narrator session в художественный markdown-черновик, дать пользователю проверить связь черновика с логом, провести разбор после главы и сохранить мягкий прогноз на 1-3 следующие главы. Stage 7 должен закрыть путь от `chapter_sessions.status="completed"` до состояния, где глава имеет draft version, пользователь принял или отложил предложения по памяти/линиям/противоречиям, а проект получил несколько вариантов дальнейшего направления без фиксации финала.

Stage 7 продолжает Stage 6: лог сессии уже хранит `session_turns`, persisted `session_suggested_actions`, `key_events`, pending `memory_proposals`, rollback/regenerate controls и статус завершения сессии. Теперь нужно использовать этот структурированный лог как вход для AI-сборки markdown, но не превращать лог в канон и не применять обновления памяти/линий без решений пользователя.

## Текущее состояние

- Репозиторий находится на ветке `v2`; новую ветку для Stage 7 не создаем по запросу пользователя.
- Parent plan определяет результат Stage 7 как: draft version из завершенной сессии, markdown storage, post-chapter review with summary/key events/memory updates/story line updates/contradictions/open questions, user decisions per suggestion, forecast with several possible directions for 1-3 chapters.
- Product docs фиксируют важные ограничения: без доступного AI черновики, разбор и прогноз открываются только для чтения; сборка/редактирование/применение решений/генерация прогноза заблокированы.
- `backend/migrations/007_v2_runtime_schema.sql` уже содержит `draft_versions`, `forecasts`, `forecast_options`, `chapters.draft_markdown`, `chapters.final_markdown`, `chapter_sessions.status` values `draft_ready` and `reviewed`.
- `backend/app/models/story_runtime.py` уже содержит `DraftVersionCreate`, `DraftVersionRecord`, `ForecastCreate`, `ForecastOptionCreate`, `ForecastRecord`, `DraftMode`, `DraftStatus`.
- `backend/app/services/story_runtime_store.py` уже умеет `create_draft_version`, `list_draft_versions`, `create_forecast`, `list_forecasts`, `create_story_line_progress`, `create_memory_proposal`, `update_memory_proposal_status`, `update_chapter`, `update_chapter_session`.
- В store пока нет helpers для получения одной draft version/forecast, обновления draft version status, выбора одного forecast option как orientation, или проектно-безопасного выбора forecast option.
- `backend/app/models/ai_actions.py` и `backend/app/ai/registry.py` уже содержат structured actions `summarize_session`, `extract_key_events`, `assemble_draft`, `edit_markdown_fragment`, `review_chapter`, `extract_memory_updates`, `update_story_lines`, `forecast_next`, `check_contradictions`.
- Отдельного backend service/API контура для `/assemble-draft`, `/draft`, `/review`, `/forecast` пока нет.
- `backend/app/api/sessions.py` покрывает session lifecycle/log/key events. `POST /api/sessions/{session_id}/complete` завершает сессию, но не запускает Stage 7.
- `backend/app/api/chapters.py` покрывает chapter CRUD and session prepare. Draft APIs from product scope are not implemented yet.
- `backend/app/api/memory.py` already exposes memory proposals and accept/reject decisions. Stage 7 should reuse this behavior instead of duplicating memory acceptance logic.
- Story line progress persistence exists, but there is no persisted review queue for story line update proposals, contradictions or open questions.
- Frontend has `entities/narrator-session` and `pages/projects/narrator-session` with log/key events/memory proposal candidates. There is no `entities/draft`, `entities/chapter-review`, `entities/forecast`, or Stage 7 route yet.
- Frontend route map currently has narrator session routes, workspace and chapter preparation. No draft assembly/review/forecast routes exist.
- `frontend/src/features/edit-chapter/ui/ChapterEditorPanel.tsx` is an older editor surface with fallback wording and legacy project-editor types. Stage 7 should not revive that component as the canonical v2 draft flow.
- `frontend/AGENTS.md` applies: desktop-only MVP, minimum viewport 1980x1024, two-column layout means 50% / 50% by default, visible frontend changes need automated checks plus Chrome DevTools QA.

## Рабочие решения

- Продолжаем в текущей ветке `v2`; branch creation is intentionally skipped.
- Stage 7 includes:
  - backend draft assembly service and API;
  - draft version reads and markdown updates;
  - lightweight draft assembly UI with session log/key events on the left and markdown/preview/actions on the right;
  - post-chapter review generation and persistent review queue for decisions;
  - applying user decisions for memory proposals, story line updates, contradictions and open questions;
  - forecast generation and orientation selection;
  - workspace/narrator CTAs from completed session to draft, from draft to review, and from review to forecast.
- Stage 7 does not include:
  - full CodeMirror 6 `OrynvaeMarkdownEditor`;
  - editor gateway, inline suggestions and transaction metadata from Stage 8;
  - final export;
  - embeddings/semantic memory retrieval;
  - automatic canon changes without user decisions;
  - manual creative fallback when AI is unavailable.
- Keep markdown as the only persisted artistic text:
  - save AI assembly output into `draft_versions.markdown`;
  - mirror current chapter draft into `chapters.draft_markdown` for simple reads/workspace summary;
  - never persist HTML, rich text JSON or editor state as canonical draft text.
- Use existing `draft_versions` instead of adding a second draft table. Add store helpers only if needed:
  - `get_draft_version(project_id, draft_version_id)`;
  - `get_latest_draft_version(project_id, chapter_id)`;
  - `update_draft_version_status(project_id, draft_version_id, status)`;
  - optional `update_draft_version_markdown` only if manual save should version in place. Prefer creating a new version for AI assembly and keeping chapter `draft_markdown` as current working copy.
- `POST /api/sessions/{session_id}/assemble-draft` is the authoritative assembly endpoint. It must:
  - require the owning project creative guard;
  - require session status `completed` or `draft_ready`;
  - load session, chapter, turns, included key events, active lines and relevant memory;
  - honor `required_event_ids`, `excluded_turn_ids`, `style_notes` and `mode`;
  - call AI action `assemble_draft`;
  - create `draft_versions` with `source_session_id`;
  - update chapter status to `draft_generated` and `chapters.draft_markdown`;
  - update session status to `draft_ready`;
  - return draft version, updated chapter/session and warnings.
- `PATCH /api/chapters/{chapter_id}/draft` updates the current markdown working copy only when AI is available. It should also mark the latest/current draft as `edited` or create a new edited draft version if that is simpler to reason about.
- `POST /api/chapters/{chapter_id}/draft/assist` uses `edit_markdown_fragment` with the full draft, selected fragment, session context, memory and key events. It returns a preview/replacement; the client applies only after user confirmation.
- Stage 7 can use a controlled markdown textarea plus preview for the assembly surface. Full CodeMirror, editor gateway and local assistant menus remain Stage 8.
- Add narrow review persistence instead of treating `review_chapter` output as an ephemeral response. Preferred shape:
  - `chapter_reviews(id, project_id, chapter_id, source_session_id, source_draft_version_id, summary, status, warnings_json, created_at, updated_at)`;
  - `chapter_review_story_line_updates(id, review_id, project_id, target_story_line_id, title, before_state, after_state, event_summary, reason, status, created_at)`;
  - `chapter_review_notes(id, review_id, project_id, note_type, title, body_json, severity, status, decision_note, created_at)`, where `note_type` covers `contradiction` and `open_question`.
- Persist memory updates from `review_chapter` through existing `memory_proposals` with `source_type="chapter_review"` and `source_id=<review_id>`. Do not create a duplicate memory proposal table.
- `POST /api/chapters/{chapter_id}/review` must:
  - require creative guard;
  - require a draft markdown source;
  - call AI action `review_chapter`;
  - create a `chapter_reviews` row;
  - persist memory proposals, story line update proposals, contradictions and open questions as pending review items;
  - update chapter status to `reviewing`;
  - return the persisted review with pending items.
- `POST /api/chapters/{chapter_id}/review/apply` must apply explicit user decisions, not the whole review blindly:
  - accepted/edited memory proposals go through existing memory service/status handling;
  - rejected/deferred memory proposals remain tracked;
  - accepted story line updates create `story_line_progress`, update the target line `current_state`, `last_progress_chapter_id`, and status when the user chose a status change;
  - unresolved story line titles require user mapping to an existing line or explicit creation of a proposed/active line from the AI suggestion;
  - contradictions/open questions can be marked resolved/deferred/rejected and can optionally point to a memory proposal decision, but they do not auto-edit canon;
  - when all required decisions are no longer pending, update session status to `reviewed` and chapter status to `completed` or keep `reviewing` if pending items remain.
- Forecast generation belongs after review, but the backend should allow a reviewed/draft-ready chapter to generate forecast as long as a draft exists. `POST /api/projects/{project_id}/forecast` should:
  - require creative guard;
  - accept `source_chapter_id`, `horizon_chapters` and `active_story_line_ids`;
  - load recent chapter summary/draft, active lines, latest story line progress, open questions and selected forecast orientation if any;
  - call `forecast_next`;
  - persist `forecasts` and `forecast_options`;
  - return 2-4 options when the model provides them, validating structured output and failing rather than inventing fallback options.
- Add forecast read/select APIs. Prefer project-scoped routes for safety even if product docs allow global forecast URLs:
  - `GET /api/projects/{project_id}/forecasts`;
  - `GET /api/projects/{project_id}/forecasts/{forecast_id}`;
  - `POST /api/projects/{project_id}/forecasts/{forecast_id}/options/{option_id}/select`.
- Forecast option selection should clear `is_selected_as_orientation` for sibling options in the same forecast and set it on the selected option. It is a creative mutation, so it is disabled without AI.
- Read APIs must remain available without AI:
  - draft versions;
  - current chapter draft markdown;
  - existing review;
  - review items and decisions;
  - forecasts and selected orientation.
- Creative state transitions and AI calls must require `runtime_status.require_creative_write(project_id)`:
  - assemble draft;
  - update draft;
  - draft assist;
  - generate review;
  - apply review decisions;
  - generate forecast;
  - select forecast orientation.
- Do not store raw prompts, raw provider responses, AI request logs or frontend debug entries in SQLite. Debug details remain JSONL-only.
- Use route names that make the flow obvious:
  - frontend draft route: `/projects/:projectId/sessions/:sessionId/draft`;
  - frontend review route: `/projects/:projectId/chapters/:chapterId/review`;
  - frontend forecast route: `/projects/:projectId/chapters/:chapterId/forecast`;
  - workspace cards should link to the highest-priority next available Stage 7 step.
- Draft assembly UI should follow the product 50% / 50% split:
  - left: log, included/excluded turns, key events, required event markers and warnings;
  - right: mode selector, assemble/regenerate action, markdown editor textarea, preview, save, proceed to review.
- Review UI should be decision-oriented, not a second editor:
  - summary at top;
  - memory proposals with accept/edit/reject/defer;
  - story line updates with target mapping and accept/reject/defer;
  - contradictions and open questions with resolve/defer choices;
  - clear pending/completed state.
- Forecast UI should show 2-4 option cards with affected lines, likely consequences and risks. Copy must make clear that selecting an option saves a soft orientation, not a fixed plan.
- Keep UI copy honest: AI suggests, user decides; draft is markdown; review does not silently change canon; forecast is not a chapter outline.

## Отклонения от исходной декомпозиции

- Chapter draft/review endpoints are project-scoped for ownership safety:
  - implemented as `GET/PATCH/POST /api/projects/{project_id}/chapters/{chapter_id}/...`;
  - not as global `/api/chapters/{chapter_id}/...` routes.
- Separate `draft_assembly.py`, `chapter_review.py`, `forecasts.py` service files were not created; Stage 7 logic lives in one cohesive `backend/app/services/stage7.py`.
- `ProjectWorkspaceRoute` itself stayed lightweight; Stage 7 workspace navigation is driven by the updated workspace summary next-step contract.
- Chrome QA is desktop-only per `frontend/AGENTS.md`: 1980x1024 and wider desktop. Mobile/tablet support is explicitly out of MVP v2 scope.

## Acceptance

- A completed narrator session can be opened from narrator/workspace CTAs into draft assembly.
- User can choose draft assembly mode and style notes.
- User can mark key events required, include/exclude session material, and assemble a markdown draft through AI.
- Assembly creates a `draft_versions` record linked to project, chapter and source session.
- Current chapter draft markdown is readable after refresh/restart.
- Chapter/session statuses move consistently through `completed` -> `draft_generated` / `draft_ready`.
- Draft text is stored only as markdown.
- User can update draft markdown only when AI is available; without AI the draft is read-only.
- Draft AI assist returns a preview/replacement and applies only after user confirmation.
- Post-chapter review can be generated from a saved draft.
- Review persists summary, memory proposals, story line update proposals, contradictions and open questions.
- User decisions are tracked per review item.
- Memory proposals do not become canon automatically.
- Accepted story line updates create progress records and update the line only after user confirmation.
- Contradictions are warnings/decisions, not automatic fixes.
- Forecast can be generated after a reviewed/drafted chapter and includes 2-4 possible directions for 1-3 chapters.
- Forecast options show affected lines, likely consequences and risks.
- Selecting a forecast option saves a soft orientation only and does not lock a plan or ending.
- Without available AI, existing drafts/reviews/forecasts remain readable, while assembly, editing, review/apply, forecast generation and orientation selection are blocked server-side and disabled client-side.
- No manual creative fallback workflow is introduced.
- No debug logs, prompts, raw provider responses or AI request logs are stored in SQLite.
- Frontend behavior is verified by automated checks plus Chrome DevTools manual QA.

## Критический анализ

- The data model already has drafts and forecasts, but not durable review decisions. Implementing review as response-only would lose pending user decisions after refresh and would not satisfy the "пользователь принимает решения по каждому предложению" requirement.
- Memory proposals already have a mature flow. Duplicating them in a review-specific table would create two acceptance paths. Better to persist review memory updates through `memory_proposals` and link them by `source_type/source_id`.
- Story line update candidates need their own pending decision surface because `story_line_progress` represents applied history, not proposals. Writing progress during review generation would silently canonize AI output.
- Contradictions and open questions are not first-class data model entities. A narrow `chapter_review_notes` table is enough for MVP decisions without inventing a full issue tracker.
- Draft editing in Stage 7 is tricky because Stage 8 owns the real CodeMirror editor. Use a modest markdown textarea/preview now and avoid building a second editor architecture.
- `PATCH /api/chapters/{chapter_id}/draft` can either mutate current chapter markdown or create a new `draft_versions` row. Mutating only `chapters.draft_markdown` is simpler but loses version history. Prefer creating clear draft versions for AI assembly and using chapter markdown as the current pointer/copy.
- Forecast selection is a mutation even though it is "soft". It must still follow read-only-without-AI rules, otherwise the app would allow creative planning in read-only mode.
- Existing session router uses global session ids with project lookup. Forecast/chapter routes must be equally careful about project ownership and not trust ids in isolation.
- Manual browser QA may require a real provider and a completed session. If provider access is unavailable, automated mocked tests must cover successful AI paths and manual QA must cover saved read-only/rendering states, with the limitation recorded.

## Риски и проверки

- Data integrity risk: applying review decisions touches memory, story lines, chapters and sessions. Service tests must verify partial decisions do not leave inconsistent statuses.
- Idempotency risk: repeated review/apply calls could create duplicate story line progress or duplicate memory proposals. Store/service should guard by review item id and status.
- Scope creep risk: draft assist can drift into full editor work. Keep fragment preview/apply minimal and defer editor gateway to Stage 8.
- Canon risk: AI review output must stay pending until the user chooses a decision.
- Context risk: assemble draft can omit important turns if exclude flags are misunderstood. Payload/tests must cover required events and excluded turns explicitly.
- Markdown risk: preview rendering must never become storage. Persist only markdown strings.
- Read-only risk: disabling frontend controls is insufficient. Backend tests must prove every creative Stage 7 mutation is blocked without AI.
- Layout risk: draft route has long logs, long markdown and Russian labels. Verify 50% / 50% columns and wrapping at 1980x1024 and wider.
- Privacy/logging risk: AI inputs/outputs may be large and sensitive. Raw prompts/responses must not be stored in SQLite; debug logs stay sanitized JSONL-only.
- UX risk: review may overwhelm the user. Group decisions by type and show pending counts rather than forcing one giant form submission.
