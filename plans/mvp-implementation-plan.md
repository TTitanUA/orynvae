# Orynvae: общий план реализации MVP

## Цель MVP

Собрать локальное web-приложение, в котором автор может подключить AI-провайдера, описать идею истории свободным текстом, создать проект с помощью AI и начать развивать мир, персонажей, сюжет, главы и канон.

MVP должен доказать основной сценарий: **идея -> AI Project Setup -> проект -> рабочее пространство истории -> первая глава и базовый канон**.

## Приоритетная очередность

Реализацию лучше вести не по отдельным экранам, а по вертикальным срезам:

1. Сначала локальный запуск, база и API-скелет.
2. Затем настройка AI-провайдера и тестовый ответ модели.
3. После этого создание проекта через AI Project Setup.
4. Затем рабочие разделы проекта: идея, мир, персонажи, сюжет.
5. Потом редактор главы и базовые AI-действия.
6. В конце MVP - канон, связность, полировка сценария и ручная проверка.

Такой порядок быстрее покажет, работает ли главная ценность продукта: автор описывает задумку, AI помогает собрать стартовую структуру, а пользователь продолжает развивать историю в одном локальном workspace.

## Этап 1. Локальный каркас приложения

**Цель:** поднять техническую основу проекта.

**Состав:**

- `backend/` на FastAPI.
- `frontend/` на Vite + React + TypeScript.
- `data/` для SQLite и локальных runtime-данных.
- `scripts/` для запуска backend и frontend одной командой.
- Базовые миграции, healthcheck API, proxy `/api`.
- Backend на `http://localhost:9001`, frontend на `http://localhost:9002`.

**Результат:** приложение локально запускается, frontend видит backend, база готова к хранению основных сущностей.

## Этап 2. Слой AI-провайдеров

**Цель:** дать приложению единый способ работы с разными моделями.

**Состав:**

- Общий интерфейс provider adapter.
- Поддержка LMStudio, Ollama, OpenAI, OpenRouter и Custom OpenAI-compatible.
- Экран настроек провайдеров.
- Проверка подключения.
- Получение списка моделей там, где это возможно.
- Выбор модели для проекта.
- Базовая поддержка streaming-ответов.
- Явная маркировка внешних провайдеров.

**Результат:** пользователь может настроить AI-провайдера, проверить его и выбрать модель для работы.

## Этап 3. Проекты и AI Project Setup

**Цель:** реализовать главный стартовый сценарий MVP.

**Состав:**

- Список проектов.
- Создание, открытие, архивирование или удаление проекта.
- Форма свободного описания идеи.
- AI-анализ идеи через выбранного провайдера.
- Предложение названия, синопсиса, жанра, тона, сеттинга, конфликта и стартовых направлений.
- Редактируемая карточка проекта перед созданием.
- Подтверждение пользователя перед сохранением.

**Результат:** пользователь может из одной идеи создать полноценный стартовый проект.

## Этап 4. Project Workspace и базовые разделы истории

**Цель:** дать автору рабочее пространство для развития проекта.

**Состав:**

- Главный экран проекта с синопсисом, настройками, провайдером и быстрыми переходами.
- Idea Lab для развития идеи, тем, мотива и конфликта.
- World Bible для мира, правил, локаций и фракций.
- Characters для карточек персонажей и отношений.
- Plot Board для арок, поворотных точек и плана глав.

**Результат:** проект можно развивать как структуру истории, а не только как текстовый документ.

## Этап 5. Chapter Editor и AI-действия

**Цель:** дать пользователю возможность писать первую главу с поддержкой AI.

**Состав:**

- Редактор глав и сцен.
- Статусы главы.
- Связанные персонажи, события и арки.
- AI-действия: продолжить сцену, переписать фрагмент, усилить конфликт, добавить описание, кратко пересказать главу.
- Сохранение AI-сообщений и предложений.
- Обработка ошибок и частичных streaming-ответов.

**Результат:** пользователь может написать и доработать первую главу внутри проекта.

## Этап 6. Базовый канон и связность

**Цель:** отделить подтвержденные факты от черновиков и AI-предложений.

**Состав:**

- Canon Facts.
- Ручное добавление фактов.
- Принятие AI-предложений в канон только после подтверждения.
- Связь фактов с персонажами, главами и событиями.
- Простая проверка очевидных противоречий.
- Базовый Timeline как часть связности проекта.

**Результат:** в проекте появляется управляемый канон, который AI не меняет без согласия пользователя.

## Этап 7. MVP-полировка и проверка пользовательского сценария

**Цель:** довести основной поток до цельного состояния.

**Состав:**

- Полировка UI как спокойного рабочего пространства автора.
- Пустые состояния, ошибки provider, состояния загрузки и streaming.
- Минимальные backend-тесты для provider layer, CRUD, миграций и AI Project Setup.
- Минимальные frontend-тесты для настройки provider, создания проекта и editor actions.
- Ручная проверка полного сценария в браузере.

**Результат:** MVP можно пройти от запуска приложения до созданного проекта, развитой структуры и первой главы.

- Шаблоны жанров.
- Профиль авторского стиля.
- Отдельные модели под planning, prose и canon checking.

## Критерий готовности MVP

MVP считается готовым, когда пользователь может:

1. Запустить backend и frontend локально.
2. Настроить AI-провайдера и выбрать модель.
3. Создать проект из свободного описания идеи.
4. Получить AI-предложение стартовой структуры проекта.
5. Отредактировать и подтвердить проект.
6. Развить идею, мир, персонажей и сюжет.
7. Написать первую главу с AI-помощью.
8. Зафиксировать важные факты в каноне.
9. Продолжить работу после перезапуска приложения без потери данных.
