# План реализации MVP v2 по этапам

## Назначение

Этот документ задает порядок реализации MVP v2 на уровне этапов. Он намеренно не является подробным task breakdown: перед началом каждого этапа нужно сделать отдельный короткий план работ по коду, используя ссылки из этого файла и актуальное состояние репозитория.

Главная линия реализации:

```text
основа данных и AI -> старт истории -> память и линии -> режим рассказчика -> черновик -> разбор -> редактор -> полная проверка
```

## Общие правила для всех этапов

- Источник продуктовой правды: [концепт v2](../docs/concept-v2.md), [MVP v2](../docs/mvp-v2.md), документы [01-16](../docs/mvp-v2/).
- Источник технической правды: [технический стек v2](../docs/technical-stack-v2.md).
- Без доступного AI любые творческие изменения заблокированы; приложение работает только на чтение.
- Markdown - единственный persisted format художественного текста.
- Debug-логи пишутся только в `logs/app-<yyyy>-<mm>-<dd>.jsonl`; в SQLite логов быть не должно.
- Для frontend-изменений после автотестов обязательна ручная проверка в браузере.

## Этап 0. Выравнивание проекта под v2

Цель: убрать старые ориентиры и зафиксировать правила разработки под новую концепцию.

Результат этапа:

- ветка `v2` используется как рабочая ветка для MVP v2;
- README ссылается только на актуальные v2-документы;
- `AGENTS.md` и `frontend/AGENTS.md` соответствуют Orynvae v2;
- старые v1-документы и ссылки на них не возвращаются.

Подтягивать детали из:

- [MVP v2](../docs/mvp-v2.md)
- [Технический стек v2](../docs/technical-stack-v2.md)
- [Критерии готовности](../docs/mvp-v2/14-acceptance-criteria.md)

## Этап 1. Runtime, данные и системные ограничения

Цель: подготовить backend/frontend основу, на которой все следующие этапы смогут безопасно сохранять состояние и блокировать творческие действия без AI.

Результат этапа:

- схема SQLite покрывает базовые сущности MVP v2;
- миграции и storage layer готовы для проектов, памяти, линий, глав, сессий, ходов, drafts и forecasts;
- есть единый guard для read-only режима без AI;
- debug logging работает по JSONL-правилу;
- markdown-поля закреплены в модели данных как canonical text storage.

## Этап 2. AI-провайдеры и action layer

Цель: превратить AI-интеграцию в типизированный слой действий, пригодный для стартового анализа, рассказчика, сборки черновика и редактора.

Результат этапа:

- provider layer поддерживает LMStudio, Ollama, OpenAI, OpenRouter и Custom OpenAI-compatible;
- доступность active provider проверяется перед творческими mutating actions;
- AI action layer имеет базовые action types и Pydantic validation;
- structured output валидируется, repair делается только как AI-запрос, без ручного fallback workflow;
- streaming/SSE контур готов для режима рассказчика.

## Этап 3. Старт истории

Цель: реализовать первый путь пользователя от свободного синопсиса к созданному проекту со стартовой памятью, линиями и точкой входа.

Результат этапа:

- экран Start Story принимает свободную идею без требования знать финал;
- AI анализирует синопсис и объясняет, как понял историю;
- пользователь подтверждает стартовую память, линии, правила расширения мира и стартовую точку;
- создается проект, готовый к первой интерактивной главе.

## Этап 4. Рабочее пространство, память и канон

Цель: дать пользователю понятное место, где видно текущее состояние истории и где AI-предложения становятся каноном только после подтверждения.

Результат этапа:

- проектный workspace показывает synopsis, текущий статус, следующий шаг и ключевые блоки;
- память истории поддерживает типы, статусы, предложения и подтверждение канона;
- предложения AI редактируются, принимаются или отклоняются;
- противоречия показываются как предупреждения, а не как автоматическое исправление.

## Этап 5. Линии истории и подготовка главы

Цель: заменить жесткий план произведения мягкими линиями и подготовкой следующей интерактивной главы.

Результат этапа:

- линии истории создаются из синопсиса и развиваются после событий;
- перед главой пользователь выбирает роль, управляемых персонажей, активные линии и примерное направление;
- подготовка главы не фиксирует финал, а задает игровую рамку для narrator mode.

## Этап 6. Режим рассказчика и лог сессии

Цель: реализовать центральный интерактивный loop, где AI ведет сцену ходами, а пользователь влияет на события.

Результат этапа:

- narrator session запускается из подготовки главы;
- пользователь может выбрать вариант действия, написать свободное действие или вмешаться как автор;
- AI отвечает ходами, сохраняются turns, suggested actions, key event candidates и memory proposal candidates;
- сессию можно поставить на паузу, продолжить и завершить;
- лог сессии читаем и пригоден для сборки черновика.

## Этап 7. Сборка черновика, разбор и прогноз

Цель: превратить прожитую сессию в художественный markdown и обновить состояние истории после главы.

Результат этапа:

- завершенная сессия собирается в draft version;
- draft хранится как markdown;
- после главы AI предлагает summary, key events, memory updates, story line updates, contradictions и open questions;
- пользователь принимает решения по каждому предложению;
- forecast показывает несколько возможных направлений на 1-3 главы без фиксации финала.

## Этап 8. Markdown-редактор и локальные AI-помощники

Цель: дать пользователю базовый редактор главы и маленьких помощников, которые работают с выделением, черновиком и контекстом проекта.

Результат этапа:

- `OrynvaeMarkdownEditor` построен поверх CodeMirror 6;
- markdown остается source of truth;
- editor gateway ограничивает агентные операции над текстом;
- AI-помощники предлагают изменения через preview/accept flow;
- без AI редактор открывается только для чтения.

## Этап 9. End-to-end hardening и демо-проход

Цель: довести MVP v2 до состояния, когда его можно пройти целиком без ручного вмешательства разработчика.

Результат этапа:

- полный browser flow проходит от настройки AI до прогноза следующей главы;
- негативный сценарий без AI проверен как read-only;
- автотесты покрывают критичные service/API/UI контракты;
- frontend вручную проверен через Chrome DevTools;
- документация обновлена под фактически реализованное поведение.

## Как делать подробный план этапа

Перед началом конкретного этапа создать отдельный рабочий план в `plans/`, например:

```text
plans/mvp-v2-stage-3-start-story.md
```

В подробном плане этапа достаточно зафиксировать:

- текущее состояние кода;
- решения по backend/frontend границам;
- список файлов или модулей, которые вероятно будут затронуты;
- тесты и ручную проверку;
- риски и вопросы, которые нельзя решить из документации.
