# План реализации MVP v2 - stage 6 Режим рассказчика и лог сессии

Дата: 2026-05-09
Ветка: `v2`
Область: backend narrator session API and services, session turn persistence, suggested actions, key events, memory proposal candidates, frontend narrator/session-log flows.
Источник: `plans/mvp-v2-implementation-plan.md`, раздел "Этап 6. Режим рассказчика и лог сессии".

## Цель этапа

Реализовать центральный интерактивный loop MVP v2: пользователь открывает подготовленную главу, запускает сессию рассказчика, отвечает действием, репликой, выбором варианта или авторской командой, получает следующий ход AI и видит сохраненный лог. Сессия должна переживать refresh/restart, уметь ставиться на паузу, продолжаться и завершаться, а ее лог должен быть достаточно структурированным для Stage 7, где из него будет собираться markdown-черновик и разбор после главы.

Stage 6 продолжает Stage 5: подготовка главы уже создает `chapter_sessions` со статусом `preparing`, связывает `chapters.session_id` и сохраняет первый AI-opening как `session_turns` запись. Теперь нужно превратить эту подготовленную рамку в рабочий narrator mode, не заходя в сборку черновика, post-chapter review, forecast или markdown editor.

## Текущее состояние

- Репозиторий находится на ветке `v2`; новую ветку для Stage 6 не создаем по запросу пользователя.
- Parent plan определяет результат Stage 6 как: запуск narrator session из подготовки главы, пользовательские действия/варианты/авторские вмешательства, AI ходы, сохранение turns, suggested actions, key event candidates, memory proposal candidates, пауза, продолжение, завершение и читаемый лог.
- Product docs фиксируют важное ограничение: без доступного AI режим рассказчика не запускается, не продолжается и не завершается; уже сохраненный лог можно только читать.
- `backend/migrations/007_v2_runtime_schema.sql` уже содержит таблицы `chapter_sessions`, `session_turns`, `key_events`, `memory_proposals`, `chapters`, `story_lines` and `story_line_progress`.
- `session_turns` уже хранит `turn_index`, `actor_type`, `turn_type`, `content`, related memory/line ids, `is_key_event`, `exclude_from_draft`; update helper для флагов пока отсутствует.
- `key_events` уже подходит для сохранения key event candidates, но отдельного статуса "candidate/accepted" нет. Для Stage 6 можно трактовать все key events сессии как редактируемые кандидаты для будущей сборки draft.
- `memory_proposals` уже подходит для memory proposal candidates через `source_type="session_turn"` или `source_type="session"` и `source_id`, но Stage 6 не должен автоматически принимать их в canon.
- Для `suggested_actions` нет отдельной persist-модели. Stage 5 сознательно оставил quick actions response-only; Stage 6 должен это исправить, иначе варианты действий исчезнут после refresh и `selected_option_id` нельзя будет проверить сервером.
- `backend/app/models/ai_actions.py` уже содержит `NarrateTurnOutput` with `narration_markdown`, `suggested_actions`, `key_event_candidates`, `memory_proposal_candidates`, `story_line_update_candidates`, `warnings`.
- `backend/app/ai/registry.py` уже регистрирует `narrate_turn` как structured action with streaming support.
- `backend/app/ai/service.py` уже умеет `execute_action` and `stream_action_events`, но streaming для structured JSON дает полезную structured часть только в конце; Stage 6 не должен строить UX, который зависит от token streaming.
- `backend/app/api/chapters.py` покрывает chapter CRUD and `session/prepare`; отдельного `/api/sessions/{session_id}` router пока нет.
- `backend/app/services/chapters.py` создает prepared session and opening turn; full lifecycle start/turns/pause/complete отсутствует.
- `story_runtime_store` уже умеет `create_chapter_session`, `get_chapter_session`, `update_chapter_session`, `create_session_turn`, `list_session_turns`, `create_key_event`, `list_key_events`, `create_memory_proposal`; update helpers for turns/key events and suggested actions are missing.
- Frontend has `entities/chapter` with `ChapterSession` and `SessionTurn` types only as part of chapter preparation result. Dedicated `entities/narrator-session` or `entities/session-log` slice does not exist.
- Frontend routes currently cover workspace, story lines and chapter preparation. There is no route for `/sessions/:sessionId/narrator` or session log.
- `ChapterPrepareRoute` shows prepared result but links only back to the project; it does not open narrator mode.
- `ProjectWorkspaceRoute` uses `workspace-summary.next_step`; backend summary currently knows planned chapters, memory attention and generic continue story, but does not surface active/paused/completed sessions as primary next steps.
- Frontend rules apply: desktop-only MVP, minimum supported viewport is documented in `frontend/AGENTS.md`; visible changes require automated checks plus Chrome DevTools manual QA.

## Рабочие решения

- Продолжаем в текущей ветке `v2`; branch creation is intentionally skipped.
- Stage 6 includes:
  - backend session lifecycle API;
  - persistence for suggested actions;
  - AI `narrate_turn` service that saves user turn, AI turn, suggested actions, key events and memory proposal candidates atomically enough for MVP;
  - read APIs for session detail, turns, log and key events;
  - flag/edit APIs for log turns and key events;
  - frontend narrator route with feed, situation/context panels, input controls and session state actions;
  - frontend log view for turns, key events and candidate memory proposals;
  - workspace and chapter-preparation CTAs that lead into narrator/log routes.
- Stage 6 does not include:
  - markdown draft assembly;
  - post-chapter review;
  - accepting memory proposals into canon;
  - applying story line progress updates after a chapter;
  - forecast;
  - markdown editor;
  - full branching/regeneration tree.
- Add a narrow migration for suggested actions rather than storing quick actions as opaque text in `session_turns`. Preferred table:
  - `session_suggested_actions(id, session_id, source_turn_id, action_index, label, action, tone, status, selected_turn_id, created_at)`;
  - `status` values can stay minimal: `suggested`, `selected`, `dismissed`;
  - `selected_turn_id` links the chosen option to the user turn when applicable.
- Add persistent per-session AI-agent settings on `chapter_sessions`: instructions, temperature, top-p and reasoning effort. These settings are session defaults for narrator turns; request-level model parameters can still override them.
- Add `source_turn_id` to `key_events` so regenerated or rolled-back narrator turns can remove their generated key-event candidates without leaving stale session artifacts.
- Keep key event candidates in existing `key_events`. Add update helper/API for title, summary, consequences, related ids and `include_in_draft`. Do not add accept/reject status unless implementation proves `include_in_draft=false` is insufficient for Stage 7.
- Keep memory proposal candidates in existing `memory_proposals` with `status="pending"`, `source_type="session_turn"`, `source_id=<ai_turn_id>`, and a clear `proposal_type`. Stage 6 displays them in log/read mode but does not apply them.
- Do not persist `story_line_update_candidates` as `story_line_progress` in Stage 6. Return/show them as AI notes only if useful. Actual line update decisions belong to Stage 7.
- Treat `NarrateTurnOutput.narration_markdown` as markdown-formatted narrator text inside structured turn content. It is not a chapter/draft markdown field and does not violate markdown-only storage.
- Define session input request types at API level:
  - `action`;
  - `dialogue`;
  - `author_command`;
  - `choice`;
  - `note`.
  Thoughts, intentions and help requests can map to `note` for MVP while the original UI label remains visible. Avoid a risky enum migration for extra turn types unless product copy truly needs separate persisted types.
- `selected_option_id` must be validated server-side against active `session_suggested_actions` for the same session. Selecting an option creates a user `choice` turn whose content is the option action, then marks the option `selected`.
- Turn index must be assigned server-side as `max(turn_index)+1` per session. The client never sends canonical indexes.
- `POST /api/sessions/{session_id}/start` transitions `preparing` or `paused` to `active`, sets `started_at` when first started, clears `paused_at`, updates chapter status to `in_session`, and returns session detail plus existing turns/suggested actions.
- `POST /api/sessions/{session_id}/turns` is the authoritative MVP endpoint. It persists the user turn, calls `narrate_turn`, persists the AI turn and candidates, and returns a complete response. Streaming can be added as a second endpoint only if it preserves the same persistence semantics.
- `PATCH /api/sessions/{session_id}/agent-settings` updates persistent narrator-agent instructions and model behavior knobs for the session.
- `POST /api/sessions/{session_id}/turns/regenerate-last` regenerates the latest AI narration with an optional user comment. It calls AI first using the prior saved user turn, then replaces the old AI tail and its generated suggested actions, key events and memory proposals only after a successful response.
- `POST /api/sessions/{session_id}/suggested-actions/regenerate` regenerates only the suggested variants for the latest narrator turn. It accepts a user `prompt` for steering the option generation, uses a dedicated structured AI action, and does not change narration, key events, memory proposals or turns.
- `POST /api/sessions/{session_id}/rollback` rolls back to a selected turn:
  - if the target is a user turn and `user_turn_mode="keep"`, the user turn remains and the following narrator response is regenerated;
  - if the target is a user turn and `user_turn_mode="redo"`, that user turn and all later turns are removed so the user can make the move again;
  - if the target is an AI narration, that narration and all later turns are removed and the narration is regenerated from the previous user turn.
- If a streaming endpoint is added, it should be a thin enhancement such as `POST /api/sessions/{session_id}/turns/stream`. It must not be the only path, and partial provider output must not create half-saved turns.
- `POST /api/sessions/{session_id}/pause` transitions `active` to `paused`, sets `paused_at`, and leaves the chapter in `in_session` or a product-consistent paused state represented by session status. Since `ChapterStatus` has no `paused`, do not add a chapter status unless needed.
- `POST /api/sessions/{session_id}/complete` transitions `active` or `paused` to `completed`, sets `completed_at`, updates chapter status to `session_done`, and returns enough log state for the UI to offer Stage 7 draft assembly later.
- Reads must remain available without AI:
  - session detail;
  - turns;
  - suggested actions;
  - log;
  - key events;
  - memory proposal candidates.
- Creative state transitions and AI calls must require `runtime_status.require_creative_write(project_id)`:
  - start;
  - submit turn;
  - pause;
  - complete;
  - editing turn/key-event flags if they change creative session material.
- If AI becomes unavailable during a session, frontend shows saved feed/log and disables continue/complete/pause where backend would block. It can link to provider settings.
- Do not store raw prompts, raw provider responses, AI request logs or frontend debug entries in SQLite. Debug details remain JSONL-only.
- Session route naming should be explicit and not collide with project routes:
  - backend router: `/api/sessions/{session_id}`;
  - frontend route: `/projects/:projectId/sessions/:sessionId/narrator`;
  - log can be a tab/query state inside the narrator route or a sibling `/log` route.
- Workspace summary should detect active/paused/preparing/completed sessions and return next-step codes such as `continue_session`, `open_session_log`, `assemble_draft` only when their target stage exists. If Stage 7 is not implemented yet, `assemble_draft` may link to log with disabled draft CTA.
- Frontend should use a new domain slice rather than overloading `entities/chapter`:
  - `entities/narrator-session` for session, turn, suggested action, key event, log API/types/queries/mutations;
  - `pages/projects/narrator-session` for route composition;
  - optional `features/play-narrator-turn` for transient input draft, selected quick action and submit state.
- The narrator UI should be a working tool surface:
  - central feed of AI/user/system turns;
  - current situation panel from latest AI turn, active lines, controlled characters and relevant memory;
  - input panel with type segmented control, free textarea, quick suggested actions and submit;
  - session controls for pause/continue/complete;
  - log tab or panel with turn flags, key events and memory proposals.
- Keep UI copy honest: suggested actions are optional, key events/memory updates are candidates, and nothing becomes canon automatically.

## Acceptance

- A prepared chapter session can be opened from chapter preparation and workspace CTAs.
- User can start a prepared narrator session, and the session/chapter status changes are persisted.
- User can choose a suggested action or write a free action/reply/author command/note.
- AI responds with one narrator turn, not a full chapter.
- Each submitted turn persists:
  - user turn;
  - AI turn;
  - suggested actions;
  - key event candidates;
  - memory proposal candidates.
- Suggested actions survive refresh and can be selected later only if they belong to the same session.
- Session feed remains readable after refresh/restart.
- Session feed updates immediately after turn submission/regeneration/rollback without requiring a browser refresh.
- User-submitted turn text appears immediately in the session feed while the narrator response is still generating.
- While the narrator response is being generated, the scene feed shows a visible generation state after the user turn area.
- Session can be paused, continued and completed.
- Session log can be opened and read turn by turn.
- Turn flags `important` / `exclude_from_draft` can be updated for draft-preparation purposes.
- Key event candidates can be listed and edited before future draft assembly.
- Memory proposal candidates are visible as pending candidates and do not become canon automatically.
- Existing confirmed memory and selected story lines are included in narrator context.
- AI does not silently change canon, apply story line progress or write chapter markdown during Stage 6.
- Without available AI, saved session/log data remains readable, but start/continue/turn submission/pause/complete/editing are blocked server-side and disabled client-side.
- No manual creative fallback workflow is introduced.
- No debug logs, prompts, raw provider responses or AI request logs are stored in SQLite.
- Frontend behavior is verified by automated checks plus Chrome DevTools manual QA.

## Критический анализ

- The biggest missing persistence piece is suggested actions. Without a table, Stage 6 cannot honestly satisfy "saved suggested actions" or validate `selected_option_id`. Adding one small migration is safer than encoding options into turn text.
- Key events are called "candidates" in product docs, but the current `key_events` table has no candidate status. For MVP, editable `key_events` plus `include_in_draft` should be enough. Adding full decision workflow would drift toward Stage 7.
- Memory proposal candidates have a mature table and decision endpoints from Stage 4, but accepting them belongs after session/draft review. Stage 6 should display source-linked pending proposals and leave canon decisions alone.
- Streaming is attractive for narrator mode, but existing structured streaming only becomes valid at the end of the model response. A non-streaming persisted turn endpoint is the reliable core. Streaming can be layered later or as a progressive enhancement if it shares the same service path.
- Current `SessionTurnType` is narrower than product input taxonomy. Mapping thoughts/help to `note` is acceptable for MVP if the UI still gives the user clear modes. Expanding the DB enum now would require a broader migration and more tests.
- Pause has no chapter-level status. Use session status as the source of truth and avoid adding `ChapterStatus="paused"` unless UI/workspace cannot represent the state otherwise.
- There is a failure-mode decision around AI errors after saving the user turn. Rolling back across SQLite writes and provider calls is awkward; keeping the user turn plus a system error note may be more transparent, but can clutter draft assembly. Decide deliberately during implementation and test it.
- `story_line_update_candidates` from `NarrateTurnOutput` are tempting to persist as progress, but that would make line updates happen during the session rather than after chapter review. Stage 6 should avoid that.
- The log must be useful for Stage 7 without becoming a markdown draft. Keep it structured and technical: turns/events/proposals, not prose assembly.
- Browser QA may require a working local provider for a real AI turn. If unavailable, automated mocked backend tests must cover successful AI behavior, and manual QA must at minimum cover read-only, saved-log and disabled-state behavior.

## Риски и проверки

- Data integrity risk: session ids are global in API, but every mutation must load the owning project and enforce project-scoped runtime guard.
- Ordering risk: turn indexes must be server-assigned and unique under concurrent clicks. Disable duplicate submits in UI and use store/helper tests for next-index behavior.
- Option ownership risk: stale or foreign suggested action ids must return 422/404 and never become a turn.
- Atomicity risk: partial persistence after AI errors can confuse logs. Tests must cover provider error behavior.
- Scope creep risk: draft assembly and post-chapter review start naturally after completion, but belong to Stage 7.
- UX risk: narrator screen can become too wide/noisy. Keep the feed central, panels scannable, and controls stable at desktop widths.
- Layout risk: long AI turns, markdown-like text, Cyrillic labels and suggested actions must wrap without overlapping buttons.
- Read-only risk: disabling frontend controls is insufficient. Backend tests must prove every creative session mutation is blocked without AI.
- Logging/privacy risk: AI action debug logging must remain sanitized JSONL-only and must not create SQLite log tables.
