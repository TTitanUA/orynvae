# Orynvae: этапы реализации MVP

## 1. Цель MVP

Цель MVP - дать пользователю возможность локально создать проект с помощью AI и начать структурировать историю без пустого документа и большой ручной формы.

Первая версия должна доказать главный продуктовый сценарий:

> Пользователь подключает локальную или совместимую AI-модель, описывает идею истории одной фразой, Orynvae помогает превратить ее в проект, а затем дает инструменты для развития мира, персонажей, сюжета и глав.

## 2. Состав MVP

### 2.1. Локальный запуск

Минимально:

- backend запускается локально;
- frontend открывается в браузере;
- backend работает на `http://localhost:9001`;
- frontend работает на `http://localhost:9002`;
- frontend проксирует запросы `/api` на backend;
- все runtime-данные хранятся в локальной папке `data/`;
- приложение не требует собственного облачного backend.

### 2.2. Настройки AI-провайдеров

MVP должен поддерживать несколько провайдеров через единый интерфейс.

Провайдеры:

- **LMStudio**
  - base URL, например `http://localhost:1234`;
  - получение списка моделей;
  - выбор модели для чата и генерации;
  - проверка доступности endpoint.

- **Ollama**
  - base URL, например `http://localhost:11434`;
  - получение списка моделей;
  - выбор модели;
  - поддержка streaming-ответов.

- **OpenAI**
  - base URL по умолчанию `https://api.openai.com/v1`;
  - API key;
  - получение списка моделей;
  - выбор модели;
  - поддержка streaming-ответов;
  - пометка как внешний провайдер.

- **OpenRouter**
  - base URL по умолчанию `https://openrouter.ai/api/v1`;
  - API key;
  - ручной ввод или загрузка списка моделей;
  - пометка как внешний провайдер.

- **Custom OpenAI-compatible**
  - произвольный base URL;
  - API key, если нужен;
  - model id;
  - тестовый запрос;
  - настройка streaming on/off.

Пользователь должен видеть, какой провайдер и какая модель используются в текущем проекте.

### 2.3. Список проектов

Функции:

- просмотр проектов;
- создание нового проекта;
- открытие существующего проекта;
- удаление или архивирование проекта;
- отображение даты последнего изменения;
- отображение выбранного AI-провайдера проекта.

### 2.4. AI Project Setup

Главный стартовый сценарий MVP.

Функции:

- ввод свободного описания истории;
- AI-анализ задумки через выбранного провайдера;
- краткий пересказ того, как AI понял идею;
- предложение 2-3 направлений развития;
- определение стартовых настроек проекта;
- уточняющие вопросы;
- редактируемая карточка проекта перед созданием;
- подтверждение создания проекта.

AI предлагает:

- рабочее название;
- краткий синопсис;
- жанр и поджанры;
- тон;
- сеттинг;
- центральный конфликт;
- главную тему;
- формат истории;
- рекомендацию по Live Mode;
- стартовые вопросы для дальнейшей разработки.

Результат:

- созданный проект;
- выбранный AI-провайдер и модель;
- подготовленные разделы: Idea Lab, World Bible, Characters, Plot Board, Timeline, Canon;
- стартовые черновики для идеи, мира и сюжета.

### 2.5. Project Workspace

Главный экран проекта.

Функции:

- обзор проекта;
- краткий синопсис;
- основные настройки;
- текущий AI-провайдер и модель;
- быстрые переходы в разделы;
- отображение текущего этапа работы;
- последние измененные главы или сущности.

### 2.6. Idea Lab

Раздел для развития первоначальной идеи.

Функции:

- хранение исходной идеи;
- расширение синопсиса;
- генерация тем и мотивов;
- варианты сеттинга;
- варианты главного конфликта;
- сравнение нескольких направлений;
- фиксация выбранной версии идеи.

### 2.7. World Bible

Раздел для мира и лора.

Функции:

- описание мира;
- правила мира;
- история мира;
- ключевые локации;
- фракции;
- магия, технологии или социальные законы;
- AI-предложения по развитию;
- базовый поиск возможных противоречий.

### 2.8. Characters

Раздел для персонажей.

Функции:

- карточка персонажа;
- имя;
- роль в истории;
- биография;
- мотивация;
- цель;
- страх;
- внутренний конфликт;
- отношения с другими персонажами;
- AI-генерация персонажа из краткого описания.

### 2.9. Plot Board

Раздел для сюжетной структуры.

Функции:

- основная сюжетная арка;
- дополнительные арки;
- поворотные точки;
- план глав;
- краткое описание глав;
- привязка персонажей, конфликтов и событий к главам.

### 2.10. Chapter Editor

Редактор для написания глав.

Функции:

- текст главы;
- статус главы;
- связанные персонажи;
- связанные события;
- AI-продолжение сцены;
- переписывание фрагмента;
- усиление конфликта;
- добавление описаний;
- краткое содержание главы.

На первом этапе проверка канона может быть простой: AI сообщает потенциальные несостыковки, но не строит сложную систему анализа.

### 2.11. Canon

Базовая система подтвержденных фактов.

Функции:

- список каноничных фактов;
- ручное добавление фактов;
- пометка AI-предложений как канон;
- связь фактов с персонажами, событиями и главами;
- простая проверка противоречий.

## 3. Основной пользовательский сценарий MVP

1. Пользователь запускает локальный backend и frontend.
2. Открывает Orynvae в браузере.
3. Настраивает AI-провайдера: LMStudio, Ollama, OpenAI, OpenRouter или Custom OpenAI-compatible.
4. Проверяет подключение и выбирает модель.
5. Открывает список проектов.
6. Нажимает **Создать проект**.
7. Вводит описание истории свободным текстом.
8. AI анализирует идею и предлагает настройки.
9. Пользователь выбирает направление и отвечает на уточняющие вопросы.
10. Пользователь подтверждает стартовую карточку проекта.
11. Система создает проект.
12. Пользователь открывает Workspace.
13. Пользователь развивает идею в Idea Lab.
14. Добавляет мир, персонажей и структуру глав.
15. Пишет первую главу в Chapter Editor.
16. Фиксирует важные факты в Canon.

## 4. AI-действия MVP

Минимальный набор AI-действий:

- проверить подключение к провайдеру;
- получить список доступных моделей;
- развить идею;
- предложить направления истории;
- определить настройки проекта;
- сгенерировать синопсис;
- предложить конфликт;
- предложить сеттинг;
- создать персонажа;
- создать фракцию;
- создать локацию;
- построить план глав;
- продолжить сцену;
- переписать фрагмент;
- усилить конфликт;
- выделить факты для канона;
- проверить очевидные противоречия.

## 6. Этапы реализации

### Этап 1. Локальный каркас

Цель: поднять приложение в архитектуре, близкой к `sd-chisel`.

Состав:

- `backend/` на FastAPI;
- `frontend/` на Vite + React + TypeScript;
- atomic design для организации frontend-компонентов;
- `data/` для SQLite и локальных файлов;
- `scripts/` для запуска backend и frontend одной командой;
- базовые миграции БД;
- healthcheck API.

### Этап 2. AI-провайдеры

Цель: дать приложению единый слой общения с моделями.

Состав:

- общий интерфейс provider adapter;
- LMStudio adapter;
- Ollama adapter;
- OpenAI adapter;
- OpenRouter adapter;
- Custom OpenAI-compatible adapter;
- экран настроек провайдеров;
- проверка подключения;
- список моделей;
- streaming-ответы в UI.

### Этап 3. Базовый MVP продукта

Цель: дать пользователю возможность создать проект с AI и начать структурировать историю.

Состав:

- список проектов;
- AI Project Setup;
- карточка проекта;
- Project Workspace;
- Idea Lab;
- базовый World Bible;
- базовые Characters;
- базовый Plot Board;
- Chapter Editor без сложной проверки канона;
- простые AI-действия.

### Этап 4. Канон и связность

Цель: превратить проект в управляемую сюжетную вселенную.

Состав:

- Canon Facts;
- извлечение фактов из глав;
- проверка противоречий;
- Timeline;
- связи между персонажами;
- связи между событиями, главами и арками;
- история изменений для важных сущностей.

### Этап 5. Live Mode

Цель: добавить интерактивное проживание сцен.

Состав:

- Live Mode-сессии;
- выбор персонажа;
- интерактивный ход сцены;
- лог действий;
- последствия решений;
- преобразование сессии в текст главы;
- подтверждение новых фактов как канон.

### Этап 6. Продвинутые функции

Цель: усилить авторский workflow и подготовить продукт к масштабированию.

Состав:

- карта мира;
- граф связей;
- совместная работа через локальный экспорт/импорт проекта;
- роли: соавтор, редактор, бета-ридер;
- экспорт в `.docx`, `.pdf`, `.md`;
- шаблоны жанров;
- профиль авторского стиля;
- расширенная аналитика структуры истории.

## 7. Нефункциональные требования MVP

- Интерфейс должен быть спокойным, плотным и рабочим.
- Пользователь должен видеть разницу между каноном, черновиком и AI-предложением.
- Все AI-предложения должны быть редактируемыми.
- AI не должен автоматически менять канон без подтверждения пользователя.
- Данные проекта должны надежно сохраняться локально.
- Внешние провайдеры должны быть явно помечены как внешние.
- Архитектура должна позволять добавить Live Mode без переписывания базовых сущностей.
- В будущем нужен экспорт проекта и глав в популярные форматы.
