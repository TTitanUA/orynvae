# 16. Редактор главы

## 1. Назначение

Редактор главы - место, где пользователь работает с художественным markdown после интерактивной сессии.

Markdown - единственный формат текста в редакторе. Глава, сцена и черновик открываются, редактируются, сохраняются и передаются AI только как markdown.

В MVP v2 редактор не должен быть профессиональным текстовым процессором. Его задача:

- показать текущий черновик;
- дать AI-зависимое редактирование;
- дать контекстные AI-действия;
- связать текст с логом сессии, памятью и линиями;
- подготовить главу к статусу "текущая принятая версия".

В MVP v2 редактор требует доступного AI для любых изменений. Если AI недоступен, редактор открывает главу только для чтения.

## 2. Отличие от режима рассказчика

Режим рассказчика отвечает на вопрос:

> Что происходит дальше?

Редактор главы отвечает на вопрос:

> Как это должно звучать как текст?

Пользователь может свободно переходить между:

- логом сессии;
- собранным черновиком;
- редактором текста;
- разбором после главы.

## 3. Макет редактора

### Верхняя панель

Показывает:

- название главы;
- статус главы;
- дату последнего сохранения;
- связанную сессию;
- активную модель;
- кнопки сохранить, собрать заново, перейти к разбору.

### Центральная область

Текстовый редактор.

Минимально:

- редактирование текста при доступном AI;
- автосохранение или явная кнопка сохранения;
- выделение фрагмента;
- поддержка абзацев;
- сохранение форматирования переносов.

Rich text, HTML и editor-specific JSON не используются как формат хранения. Preview может рендерить markdown в HTML, но HTML не становится источником правды.

### Боковая панель

Показывает:

- ключевые события сессии;
- связанные линии;
- важные факты памяти;
- предложения AI;
- предупреждения канона.

## 4. Статусы главы в редакторе

### Черновик AI

Текст создан из сессии и еще не правил пользователь.

### Редактируется

Пользователь внес изменения.

### Требует разбора

Текст содержит новые факты или изменения, которые нужно проверить.

### Принятая версия

Пользователь сохранил текст как текущую рабочую версию главы.

### Закрыта

Глава считается завершенной для текущего этапа, но может быть открыта позже.

## 5. AI-действия редактора

### Для всей главы

- проверить связность;
- сделать краткое резюме;
- найти новые факты;
- предложить название;
- улучшить ритм;
- сократить;
- расширить;
- сделать тон более мрачным, теплым, напряженным и т.д.

### Для выделенного фрагмента

- переписать проще;
- переписать выразительнее;
- усилить конфликт;
- усилить эмоцию;
- улучшить диалог;
- добавить атмосферу;
- сократить;
- объяснить, почему фрагмент слабый;
- предложить 3 варианта.

### Для диалога

- сделать реплики естественнее;
- добавить подтекст;
- разделить голоса персонажей;
- убрать прямое объяснение;
- добавить паузы и реакцию.

## 6. Связь с логом сессии

Редактор должен помогать не потерять прожитые события.

Боковая панель показывает ключевые события:

- вошло в текст;
- не вошло;
- изменено;
- исключено пользователем.

Пользователь может:

- вставить событие в текст;
- пометить событие как неважное;
- попросить AI вплести событие в текущий абзац.

## 7. Связь с памятью истории

Редактор должен находить:

- новые факты;
- новые имена;
- новые места;
- изменения отношений;
- нарушения канона.

Но он не должен автоматически менять память.

При сохранении или переходе к разбору AI может предложить:

> В тексте появился факт: "архивариус знал отца героя". Добавить в память истории?

Пользователь выбирает:

- добавить как канон;
- добавить как черновик;
- изменить;
- отклонить.

## 8. Связь с линиями истории

Редактор показывает, какие линии затронуты в главе.

Примеры:

- линия тайны продвинулась через новый след;
- линия отношений получила конфликт;
- линия угрозы усилилась;
- линия героя не изменилась.

Если глава не продвигает ни одну активную линию, это не ошибка. AI может мягко заметить:

> Глава работает как атмосферная сцена. Если хочешь, можно добавить маленькое продвижение тайны или отношений.

## 9. Сохранение

MVP должен надежно сохранять:

- markdown главы;
- статус;
- дату изменения;
- связь с draft version;
- связь с сессией;
- правки пользователя, сделанные в AI-доступном режиме.

Автосохранение желательно, но можно начать с явной кнопки, если она надежна и заметна.

## 10. Пересборка черновика

Пользователь может пересобрать текст из лога.

Перед пересборкой нужно предупредить:

- текущие правки могут быть заменены;
- можно создать новую версию, а не перезаписать.

MVP-подход:

- создавать новую DraftVersion;
- не удалять предыдущую версию.

## 11. Версии

Минимальная версия в MVP:

- текущий текст;
- предыдущие AI-сборки как DraftVersion.

Не обязательно делать полноценный version control UI.

Достаточно:

- список версий;
- дата;
- режим сборки;
- открыть;
- сделать текущей.

## 12. Ошибки

### AI-действие не удалось

Текст не должен теряться.

Показать:

- ошибку;
- повторить;
- попробовать другой режим;
- открыть настройки AI.

### Конфликт автосохранения

Если появится в будущем, показывать две версии. В MVP лучше избегать сложного совместного редактирования.

## 13. Критерии готовности

Редактор главы готов для MVP, если:

- можно открыть черновик;
- можно редактировать markdown;
- можно сохранить;
- можно применить AI-действие к выделенному фрагменту;
- можно видеть ключевые события сессии;
- можно перейти к разбору после главы;
- текст не теряется при ошибке AI;
- новые факты не попадают в канон без подтверждения;
- при недоступном AI редактор работает только в режиме чтения.
