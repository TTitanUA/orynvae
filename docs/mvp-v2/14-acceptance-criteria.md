# 14. Критерии готовности

## 1. Главный end-to-end сценарий

MVP v2 считается готовым только если можно вручную пройти полный сценарий в браузере:

1. Открыть приложение.
2. Настроить AI-провайдера.
3. Создать проект из синопсиса.
4. Проверить AI-анализ.
5. Подтвердить стартовую память.
6. Выбрать правила расширения мира.
7. Выбрать стартовую точку.
8. Открыть рабочее пространство проекта.
9. Подготовить первую главу.
10. Запустить режим рассказчика.
11. Сделать несколько ходов пользователя.
12. Завершить сессию.
13. Посмотреть лог.
14. Собрать черновик главы.
15. Провести разбор после главы.
16. Подтвердить обновления памяти.
17. Обновить линии истории.
18. Получить прогноз следующей главы.

Отдельно нужно проверить негативный сценарий: если AI недоступен, пользователь может читать существующие данные, но не может создавать, редактировать или продолжать историю.

## 2. Критерии по старту истории

- Пользователь может ввести короткую идею.
- Финал не требуется.
- AI возвращает пересказ идеи.
- AI выделяет стартовые элементы памяти.
- AI предлагает стартовые линии.
- AI предлагает стартовые точки.
- Пользователь может редактировать предложения.
- Проект создается без заполнения больших форм.
- После создания есть понятный CTA к первой главе.
- Без доступного AI старт истории заблокирован.

## 3. Критерии по памяти истории

- Элементы памяти имеют типы и статусы.
- Канон визуально отличим от черновика и AI-предложения.
- Пользователь может подтвердить, изменить или отклонить предложение.
- После сессии появляются предложения обновлений.
- AI использует подтвержденный канон в режиме рассказчика.
- Возможные противоречия показываются как предупреждения.
- Без доступного AI память открывается только для чтения.

## 4. Критерии по линиям истории

- Линии извлекаются из синопсиса.
- Пользователь может принять, изменить или отклонить линию.
- Линии видны в проекте.
- Перед главой можно выбрать активные линии.
- После главы AI предлагает обновления линий.
- Прогноз строится с учетом линий.
- Линии не требуют заранее знать финал.
- Без доступного AI линии открываются только для чтения.

## 5. Критерии по режиму рассказчика

- Пользователь выбирает роль перед главой.
- AI начинает сцену с учетом стартовой точки.
- AI отвечает ходами, а не генерирует целую главу.
- Пользователь может выбрать вариант действия.
- Пользователь может написать свободное действие.
- Пользователь может использовать авторскую команду.
- Ходы сохраняются в лог.
- Сессию можно поставить на паузу.
- Сессию можно продолжить.
- Сессию можно завершить.
- Важные новые элементы не становятся каноном автоматически.
- Без доступного AI режим рассказчика не запускается и не продолжается.

## 6. Критерии по сборке черновика

- Из завершенной сессии можно создать черновик.
- Черновик сохраняет ключевые решения пользователя.
- Пользователь может выбрать режим сборки.
- Пользователь может редактировать черновик.
- AI-инструменты работают с выделенным фрагментом.
- Черновик связан с главой и сессией.
- После черновика доступен разбор.
- Без доступного AI черновик открывается только для чтения.

## 7. Критерии по разбору после главы

- AI показывает резюме.
- AI выделяет ключевые события.
- AI предлагает новые факты.
- AI предлагает новые или измененные элементы памяти.
- AI предлагает обновления линий.
- AI показывает открытые вопросы.
- AI показывает возможные противоречия.
- Пользователь принимает решения по каждому предложению.

## 8. Критерии по прогнозу

- Прогноз появляется после главы.
- Есть 2-4 варианта направления.
- Каждый вариант показывает, какие линии затрагивает.
- Каждый вариант показывает возможные последствия.
- Возможные финалы показываются как гипотезы.
- Выбор варианта не фиксирует жесткий план.

## 9. Критерии по AI-провайдерам

- Можно настроить локальный провайдер.
- Можно настроить внешний провайдер.
- Можно проверить подключение.
- Можно выбрать модель.
- Активная модель видна в проекте.
- Внешний провайдер явно помечен.
- Ошибки подключения понятны.

## 10. UX-критерии

- Пользователь всегда видит следующий разумный шаг.
- Термины объясняются простым языком.
- Большие формы не являются обязательным стартом.
- AI-предложения редактируемы.
- Интерфейс не заставляет заранее знать финал.
- Состояния загрузки и ошибок понятны.
- Сохранение состояния видно.
- Незавершенная сессия не теряется.
- Без AI приложение ясно показывает режим чтения и не предлагает ручной творческий fallback.

## 11. Нефункциональные критерии

- Данные проекта сохраняются локально.
- Перезапуск приложения не теряет проекты.
- Перезапуск приложения не теряет незавершенную сессию.
- Backend и frontend запускаются проектным dev-скриптом.
- Ошибки backend не показываются пользователю как stack trace.
- AI-запросы не блокируют весь интерфейс.
- Streaming не обязателен для всех провайдеров, но UI должен корректно работать с ним и без него.
- Любые изменения истории, памяти, линий, сессий, черновиков и прогнозов требуют доступного AI.
- При включенном `DEBUG` диагностические события пишутся в `logs/app-<yyyy>-<mm>-<dd>.jsonl`.
- В SQLite нет таблиц и записей для debug logs или AI request logs.
- Запросы к `/api/debug/logs` не попадают в debug logs.

## 12. Проверка frontend

Для frontend-изменений после автоматических тестов нужно:

- запустить локальную среду через проектный dev launcher;
- открыть приложение в браузере;
- пройти основной пользовательский сценарий;
- проверить видимые layout-проблемы;
- проверить консоль браузера;
- проверить network ошибки;
- убедиться, что интерактивный режим не ломает экран на длинных ответах.

## 13. Минимальная демо-история для проверки

Для ручного QA можно использовать идею:

> В городе, где люди покупают чужие воспоминания, обычный курьер находит воспоминание о собственной смерти и понимает, что кто-то уже прожил его будущее.

Проверка должна подтвердить:

- AI выделяет город, курьера, рынок воспоминаний, воспоминание о смерти;
- AI предлагает тайну;
- AI предлагает линию героя и линию угрозы;
- стартовая сцена может начаться в архиве, на рынке или во время доставки;
- режим рассказчика позволяет пользователю решить, читать ли воспоминание сразу;
- после сцены память предлагает новые факты;
- прогноз не фиксирует финал.

## 14. Definition of Done MVP v2

MVP v2 готов, если:

- полный end-to-end сценарий проходит без ручного вмешательства разработчика;
- основные данные сохраняются и восстанавливаются;
- пользователь может получить первый черновик главы из интерактивной сессии;
- канон и черновики различаются;
- линии истории помогают продолжению;
- AI-провайдеры работают через единый интерфейс;
- режим чтения без AI проверен;
- debug JSONL logging проверен без записи в БД;
- frontend вручную проверен в браузере;
- документация MVP v2 отражает реализованный функционал или явно помечает отклонения.
