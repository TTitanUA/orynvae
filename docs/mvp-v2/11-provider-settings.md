# 11. AI-провайдеры и приватность

## 1. Назначение

AI-провайдеры - техническая основа всех AI-функций MVP v2.

Приложение должно поддерживать локальные и внешние модели, но всегда ясно показывать пользователю, куда отправляются данные.

Без доступного AI-провайдера Orynvae v2 работает только в режиме чтения. Приложение не должно предлагать ручные fallback-сценарии создания или редактирования истории.

## 2. Провайдеры MVP

### LMStudio

Поля:

- base URL;
- model id;
- streaming on/off;
- test connection.

Обычно:

- `http://localhost:1234`

### Ollama

Поля:

- base URL;
- список моделей;
- выбранная модель;
- streaming on/off;
- test connection.

Обычно:

- `http://localhost:11434`

### OpenAI

Поля:

- base URL;
- API key;
- список моделей;
- выбранная модель;
- streaming;
- test connection.

По умолчанию:

- `https://api.openai.com/v1`

UI должен помечать как внешний провайдер.

### OpenRouter

Поля:

- base URL;
- API key;
- model id;
- список моделей, если доступен;
- streaming;
- test connection.

По умолчанию:

- `https://openrouter.ai/api/v1`

UI должен помечать как внешний провайдер.

### Custom OpenAI-compatible

Поля:

- название;
- base URL;
- API key, если нужен;
- model id;
- streaming on/off;
- test connection.

## 3. Экран настроек

### Основные блоки

- список сохраненных провайдеров;
- форма редактирования;
- активный провайдер по умолчанию;
- проверка подключения;
- выбор модели;
- предупреждения приватности;
- тестовый AI-запрос.

### Статусы

- не настроен;
- настроен, но не проверен;
- доступен;
- недоступен;
- ошибка авторизации;
- модель не выбрана;
- streaming недоступен.

## 4. Приватность

### Локальный провайдер

UI должен объяснять:

> Запросы отправляются на локальный сервер модели. Данные не уходят во внешний AI-сервис через Orynvae, но локальный сервер может иметь свои настройки.

### Внешний провайдер

UI должен объяснять:

> Для этой модели текст истории, память и контекст сцены отправляются внешнему AI-провайдеру.

Это предупреждение должно быть видно:

- при настройке провайдера;
- при выборе внешней модели для проекта;
- перед первым AI-запросом в проекте, если пользователь еще не подтверждал.

## 5. Выбор провайдера на проект

Проект должен хранить:

- активный провайдер;
- активную модель;
- настройки streaming;
- дату последней проверки.

Пользователь может поменять провайдера проекта.

При смене провайдера UI должен предупредить:

- стиль ответов может измениться;
- внешняя модель может получить контекст истории;
- локальная модель может быть слабее в длинном контексте.

## 6. Контекст AI-запроса

Для MVP важно не отправлять лишнее.

Каждый AI-запрос должен собирать только нужный контекст:

- анализ синопсиса: синопсис и настройки пользователя;
- ход рассказчика: текущая сессия, активные факты, линии, участники;
- сборка черновика: лог сессии и релевантная память;
- разбор главы: черновик, ключевые события, память;
- прогноз: последние главы, линии, открытые вопросы.

## 7. Ошибки AI

### Ошибка подключения

Показать:

- адрес провайдера;
- краткую причину;
- кнопку повторить;
- ссылку на настройки.

### Ошибка авторизации

Показать:

- что ключ не принят;
- не показывать ключ целиком;
- предложить обновить API key.

### Ошибка модели

Показать:

- model id;
- совет выбрать другую модель;
- возможность обновить список моделей.

### Прерывание streaming

Показать:

- частичный ответ, если он пригоден;
- кнопку продолжить;
- кнопку повторить.

Если модель полностью недоступна, "продолжить" не должно означать продолжение без AI. Пользователь может только повторить запрос после восстановления подключения или вернуться к чтению сохраненных данных.

## 8. Хранение секретов

MVP должен хранить API key локально.

Минимально допустимо:

- локальная БД или конфиг;
- не показывать ключ после сохранения целиком;
- дать кнопку удалить ключ.

В будущем можно улучшить через системное хранилище секретов.

## 9. Критерии готовности

AI-провайдеры готовы для MVP v2, если:

- можно добавить и проверить провайдера;
- можно выбрать модель;
- проект знает активный провайдер;
- внешние провайдеры явно помечены;
- основные AI-функции используют выбранного провайдера;
- ошибки понятны пользователю;
- streaming работает в режиме рассказчика, если поддерживается;
- при недоступном провайдере приложение переходит в режим чтения, а не в ручной творческий режим.
